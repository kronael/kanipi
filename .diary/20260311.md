## Media Routing Path Fix (evening)

Fixed critical bug where media was saved to wrong folder during routing:

**Problem**: When a message with media arrived at a JID with routing rules (e.g.,
telegram:1112184352 → atlas/support), media was saved to the default folder
(groups/atlas/media/) but the container mounted the routed target folder
(groups/atlas/support/), causing 404s on media paths.

**Root cause**: `enqueueEnrichment()` was called in `onMessage` callback using
the default route folder BEFORE routing rules were evaluated. Routing happened
later in the message loop, but media was already saved to wrong location.

**Fix**: Resolve routing immediately in `onMessage` callback before calling
`enqueueEnrichment()`:

```typescript
const routes = getRoutesForJid(msg.chat_jid);
const targetFolder = resolveRoute(msg, routes) || group.folder;
enqueueEnrichment(msg.id, targetFolder, attachments, download);
```

This ensures media is saved to the final routed target folder, not the default.

## ENTRYPOINT Inheritance Bug Fix (22:00)

**Problem**: Agent containers crashed with `SyntaxError: Unexpected token ':'` when
trying to parse JSON stdin. The bash entrypoint script was being executed as Node.js code.

**Root cause**: `node:22-slim` base image sets `ENTRYPOINT ["docker-entrypoint.sh"]`
which was being inherited by our Dockerfile. When we specified a command in docker args,
it was appended to the inherited entrypoint instead of replacing it.

**Fix**: Added `ENTRYPOINT []` to `container/Dockerfile` line 72 to clear the inherited
entrypoint. Now the gateway can supply the command directly via docker args.

## Schedule Task Schema Documentation (22:01)

**Problem**: Atlas agent created scheduled tasks incorrectly, using `prompt` (agent mode)
instead of `command` (raw bash mode) for simple maintenance scripts like refresh-repos.sh.

**Root cause**: The `schedule_task` action had no field-level descriptions in its Zod schema.
Agents only saw action description "Schedule a recurring or one-time task" with bare field
names/types. No explanation that `prompt` spawns full Claude agents vs `command` runs bash directly.

**Fix**: Added comprehensive `.describe()` calls to all ScheduleTaskInput fields in
`src/actions/tasks.ts`, explaining:

- `prompt`: "Agent prompt - spawns a full Claude Code agent with reasoning, tool use..."
- `command`: "Bash command to run directly without agent ceremony. Use for simple maintenance scripts..."
- When to use each mode
- Example values for schedule_value

## Session Transcript Reading Nudges (22:08)

**Problem**: Agents weren't reading previous session transcripts on startup even though
`.jl` files existed in `~/.claude/projects/-home-node/`. They claimed "without access
to the full session history" instead of using Read tool.

**Root cause**: All session reading instructions used weak/conditional language:
"if needed", "if the user asks", "if the user references". Agents interpreted this
as optional behavior.

**Fix**: Strengthened wording across all CLAUDE.md files and skills:

1. **Root CLAUDE.md** - Changed "If the user references" to "ALWAYS read" with explicit
   instructions to use Glob + Read on `.jl` files
2. **container/CLAUDE.md** - Added numbered mandatory steps: check diary, read transcript,
   use it. Added CRITICAL warning about never claiming "no access to history"
3. **container/skills/self/SKILL.md** - Added "MANDATORY workflow on new session" section
   with bash example commands
4. **container/agent-runner/CLAUDE.md** - Added "MANDATORY on new session" heading with
   3-step workflow
5. **src/index.ts** - Added comment explaining that gateway injects session IDs and agents
   are instructed to read the .jl files
