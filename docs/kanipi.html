Kanipi v1

modular agent deployment on NanoClaw

Status: v0.7.0 released. Kanipi is a NanoClaw fork — a multi-tenant, multi-channel Claude agent gateway. Five channels in production: Telegram, Discord, WhatsApp, Email, Web. Runtime tested with real bots and users.
TL;DR
The ideal agent exists in pieces: NanoClaw's container isolation, OpenClaw's routing brain, nobody's credential injection. Kanipi assembles them. Started as ~500 LOC on NanoClaw's ~2,600 foundation, now grown to 9,200+ LOC with 5 channels, voice transcription, MCP sidecars, diary memory, forward metadata, file commands, action registry, and a comprehensive test suite. V2 adds audience agents and knowledge systems. V3 (Arizuka) adds cross-agent collaboration. 40 source files, 9,272 LOC (+27 test files, 7,809 LOC).

The Problem

Three products. Five channels. One infrastructure gap.

Elza — Support

The dev group already has a technically knowledgeable colleague — one that knows your architecture, your tradeoffs, and what a milestone actually means. CLAUDE.md is the hire. A generic assistant would be noise. This one earns its place because it knows the project.

Eliza — Evangelist

Drafts posts and threads for X with your voice. V2 middleware will intercept every post tool call before it reaches the API — writing to a pending queue instead. No draft publishes without a human approval record. “Always ask first” won't be a prompt rule. It'll be a route.

Envoy

Monitors IMAP, triages incoming messages, drafts replies. Email channel is shipped in V1 — IMAP IDLE for real-time inbox, SMTP with threading for replies.

Multiple agents on different channels means three constraints that must hold simultaneously:

  • Isolation — each agent runs in its own container. Eliza cannot read Takopipi's API keys.
  • Routing — a Telegram message must reach the right agent. Different chats, different agents, one bus.
  • Secret injection — the host injects API keys at call time. The agent never holds credentials in process memory.

Once you need all three, you also need: per-agent CLAUDE.md (different personality per agent), config instead of hardcoded constants, and systemd deployment with one service per instance.

The Gap

Five systems studied, none solves all three. OpenClaw has routing (24 channels, binding cascade) but zero isolation — SDK on host, tools bypass sandbox. NanoClaw has container isolation but hardcoded constants and no routing — every message goes to one agent. Muaddib has QEMU micro-VMs and host/guest secret split but max 8 concurrent and no channel routing.

The architecture comparison concludes: the ideal agent exists in pieces. NanoClaw's host process has zero SDK imports — the container is a plugin slot. That's the foundation. Extract OpenClaw's binding cascade (~200 LOC), add config parsing (~100 LOC), credential middleware (~50 LOC), channel adapters (~150 LOC). The gap is ~500 LOC of new components on top of NanoClaw's ~2,600.

Systems Compared · 13 Patterns That Work

The Foundation: NanoClaw

NanoClaw solves multi-tenancy via isolation. Kanipi adds multi-tenancy via routing. Together: each agent runs in its own container, and messages find the right agent automatically.

NanoClaw is a ~2,600 LOC host orchestrator with zero SDK imports. The host process doesn't know or care what runs inside the container — Claude Agent SDK, a Python script, anything that reads IPC files. The container is a plugin slot.

What it gives us
  • Container isolation per agent (Docker, not opt-in)
  • Claude Agent SDK + browser automation inside each container
  • Signal-driven IPC: gateway sends SIGUSR1 after writing IPC file; agent wakes immediately (500ms fallback)
  • Subagent teams — each sub-agent gets its own container + IPC directory, no lateral communication
  • Session persistence via per-group CLAUDE.md files
  • Scheduled tasks (cron/interval/once)
  • 5-layer secret isolation: stdin → ephemeral disk → SDK-only → bash scrubbing → memory scrubbing
What's missing
  • Hardcoded constants instead of config file
  • WhatsApp only — no Telegram, Discord, Email, Web
  • No routing — every message goes to one agent
  • No credential injection — secrets passed as env vars, live in process memory
  • No per-agent personality — one CLAUDE.md for all
  • No auto-registration — manual setup per chat
  • No shared memory across agents — per-group markdown, no cross-tenant state

Each agent container receives a structured workspace namespace:

/workspace/self replaced /workspace/project: exposes the full kanipi source and all groups (not just main). /workspace/web/pub/ is publicly served; /workspace/web/priv/ requires auth.

Skills System

container/skills/ is seeded to ~/.claude/skills/ inside each container on first spawn for a group. Includes kanipi skills plus development skills bundled from kronael/tools (bash, go, python, typescript, etc.). A CLAUDE.md is seeded alongside.

character.json

Agent identity in container/character.json (ElizaOS-style: bio, topics, adjectives, style, messageExamples). Fields are randomized per query. Per-instance overrides via /workspace/global/character.json are merged at load time. Replaces SOUL.md.

Migration System

container/skills/self/MIGRATION_VERSION tracks applied version. Numbered files in container/skills/self/migrations/ describe changes. The /migrate skill syncs all groups from canonical source when version advances.

V1: Five Channels Shipped

Everything NanoClaw has, plus config, five channels, group routing, voice transcription, file sending, forward metadata, file commands, MCP sidecars, diary memory, capability introspection, action registry, and security hardening.

What V1 Ships
1
.env + env vars
All config via .env in instance data dir or environment variables. Env var wins when both are set. Channels enabled by presence: TELEGRAM_BOT_TOKEN, DISCORD_BOT_TOKEN, EMAIL_IMAP_HOST, WhatsApp auth dir. No YAML, no TOML — flat env is the config.
2
Five channel adapters
Telegram via grammy (polling). Discord via discord.js (gateway). WhatsApp via baileys (inherited from NanoClaw). Email via imapflow (IMAP IDLE real-time) + nodemailer (SMTP threading). Web (Slink — the web channel handler) — authenticated HTTP channel with HMAC JWT auth, rate limiting, SSE streaming. Markdown→HTML formatting, typing indicators, chunked replies (Telegram's 4096-char limit handled automatically). Built-in /chatid and /ping commands across all channels.
3
Group routing + delegation
Groups registered via CLI with folder mapping. First group defaults to folder=main. Subsequent groups use trigger mode (agent responds only when mentioned). Agents can delegate prompts to child groups via delegateToChild IPC action — explicit, not automatic. Unregistered chats silently dropped.
4
Action registry
Unified action system with Zod schema validation. Agents request actions via IPC, gateway validates and executes. Domain handlers: messaging (send text, send document), tasks (schedule, cancel), groups (delegate to child), session (manage state). Authorization enforced per action — agents can only act on their own group.
5
Voice transcription + file sending
Voice messages transcribed via Whisper sidecar before delivery to agent. Agents send files back to channels via sendDocument IPC action — path-safe (must be under group dir). Media type detection via magic bytes (file-type). Configurable per instance: VOICE_TRANSCRIPTION_ENABLED, MEDIA_ENABLED.
6
MCP sidecars
Per-group MCP server configuration. Sidecar processes run alongside agent containers, exposed as MCP tools. Gateway manages sidecar lifecycle — start, health check, cleanup. Stale sidecar configs purged automatically on group remove/disable. Whisper transcription is itself an MCP sidecar.
7
Diary memory
Agent-written daily notes in /workspace/group/diary/YYYYMMDD.md. YAML frontmatter summary injected on session start (two most recent files, with relative time: “today”, “yesterday”, “3 days ago”). PreCompact hook nudges agent to write entries before context compression. The diary IS the task log — tasks appear and change state through entries.
8
Task scheduler
SQLite-backed scheduler: cron, interval, and once modes. Agents request scheduled tasks via IPC. Two context modes: group (shares conversation history) or isolated (fresh context per run). Run log with per-task status and last result.
9
Circuit breaker + error handling
Per-group consecutive failure counter. After 3 failures, circuit opens — no more automatic retries. New user message resets the breaker. Agent errors advance the cursor and notify the user to retry manually. No exponential backoff loops on permanent failures.
10
IPC auth + mount security
Groups cannot impersonate each other. Each container can only send messages to its own registered JID. Host-side allowlist (~/.config/nanoclaw/mount-allowlist.json) controls bind mounts — the allowlist itself is never mounted (tamper-proof from agents). Per-group containerConfig sets additional mounts and container timeout.
11
Per-agent personality
Each group gets its own CLAUDE.md, character.json, and skills. Different agents, different chats, different personalities. System messages formatted per channel with agent name and context.
12
Forward metadata
Forwarded messages and reply-to context extracted per channel and rendered as nested XML tags in the agent prompt. Telegram: full forward origin (user, hidden, chat, channel). WhatsApp: forward flag. Discord: fetchReference() for replies. Reply text truncated to 100 chars.
13
File commands
/put, /get, /ls — bidirectional file transfer between chat users and group workspace. Path safety with deny globs, symlink escape protection, atomic writes. Disabled by default (FILE_TRANSFER_ENABLED).
14
Capability introspection
Gateway writes .gateway-caps TOML manifest to group directory before each agent spawn. Agent reads it to accurately answer capability questions (voice, video, media limits, web host). Agent-writable .whisper-language configures transcription languages.
15
Hot-patching
Gateway mounts agent-runner TypeScript source into containers. Entrypoint recompiles with bunx tsc at runtime — changes to agent behavior deploy without rebuilding the container image. Dockerfile validates compilation at build time.

V2: Audience Agents

V1 serves the operator. V2 serves many users — agents that know who they're talking to, what they know, and what they've discussed before. Topic routing, per-user context, knowledge systems.

Topics — Thread-Level Routing
1
#topic routing
Route messages within a group to different agents by topic or thread. Topic ID as JID segment (builds on the worlds router). Glob pattern matching. Same group, different conversations, different handling.
2
Session resume tokens
Embed session ID in reply footer, extract on next message, pass --resume <id> for explicit continuation. Thread-level granularity within groups — the takopi pattern.
Memory Layers
1
Diary (shipped in V1)
Daily markdown notes with YAML frontmatter. The raw input layer. Two most recent files injected on session start with relative time labels.
2
Work (current task state)
Ephemeral work.md describing active task, blockers, next steps. Overwritten each time, not appended. The /work skill manages it. Most volatile layer — diary captures the history that work.md cycles through.
3
Episodes (weekly/monthly)
Scheduled aggregation of diary entries into episode summaries. Compresses daily notes into longer-term patterns and milestones.
4
Facts (permanent)
Conceptual knowledge distilled from episodes. Entities, relationships, recurring patterns. Inspired by ElizaOS facts repository and brainpro's MEMORY.md.
Agents + Workflows
1
@agent personas
Different personas and configs per agent. Currently group = agent (1:1). V2 patterns: topic-based, skill-based, model-based agent selection.
2
Workflow pipelines
Multi-step: enrich → prompt → LLM call → transform → route. Agent orchestrates or gateway pre-defines. Deterministic pipelines without LLM in the loop.
3
IPC → MCP migration
Replace file-based IPC with MCP over Unix socket. Structured protocol, bidirectional, typed tool calls. File IPC remains as fallback.
Secret Boundary

The secret boundary problem: SDK-on-host (OpenClaw, Muaddib) keeps secrets safe but tools bypass the sandbox. SDK-inside-container (NanoClaw) isolates tools but secrets must cross in — five layers of protection, yet the key IS in process memory. V2's MCP proxy enables credential injection at the protocol level: the agent calls a tool, the gateway adds the API key before forwarding. The credential never enters the container's process memory. Full solution requires network isolation (--network none) to prevent exfiltration.

V3: Arizuka

Arizuka (蟻塚, Japanese: anthill). V1 and V2 give one agent per channel. Arizuka is the system where agents collaborate across containers.

Ant Colony
1
Cross-agent messaging
Agents in separate containers send messages to each other via the IPC bus. Today subagents are hierarchical (parent → child). Arizuka adds peer-to-peer: agent A requests agent B without going through a parent.
2
Shared memory
NanoClaw uses per-group CLAUDE.md files — transparent and auditable, but no cross-agent state. Arizuka needs a shared store with scoped permissions. The multi-tenant memory gap is the hardest open problem: the simplest system (CLAUDE.md) and the most complex (Muaddib's 3-tier chronicles) both converge on human-readable markdown — but neither shares across agents.
3
Multi-host bus
Today the IPC bus is filesystem-based (one machine). Multi-host requires a network transport layer — the bus abstraction already exists, the transport is the gap.
4
Web dashboard
Observe all agents, their state, costs, token usage, message flow. Built on the same IPC bus as everything else.
5
Component crystallization
Router, action registry, IPC bus extract into standalone packages. Today they live in ~38 source files. Extraction happens when the interfaces stabilize through use.
V3 is designed but not implemented. The components above will crystallize from V1/V2 usage — not from spec documents.
